# Hi-Voicer 模型说明

## 录音和转写不是同一件事

录音只负责采集麦克风或系统声音。真正把声音变成文字的是 ASR 引擎和 ASR 模型。Hi-Voicer 的语音输入和文件转写会共用同一类 ASR 模型，只是使用场景不同：

- 语音输入：麦克风音频持续送入 Silero VAD，检测到一句结束后立即用 Sherpa-ONNX 离线识别该段并输入文字。
- 文件转写：长音频，优先稳定性、准确率和可保存结果。

## CapsWriter 的模型处理思路

CapsWriter-Offline 不是“只下载模型就能识别”。它的本地识别链路是：录音模块生成 WAV，ASR 引擎读取音频，ASR 引擎加载模型文件，再输出文字。不同模型有不同适配器和推理后端。

SenseVoiceSmall 也需要引擎。它只是自己具备标点能力，不代表不需要接入引擎。

## 当前模型列表策略

当前正式列表只保留两个随安装包交付、已接入本地推理链路的模型。应用运行时不提供模型下载入口。

SenseVoice 模型来自固定的 Hugging Face 修订版，Qwen3-ASR 与 Silero VAD 来自 Sherpa-ONNX 官方模型发布。两种 ASR 都统一使用 Sherpa-ONNX CPU 推理，不携带 CUDA 或 Vulkan 组件。

## 当前预设

| 模型 | 状态 | 用途 |
| --- | --- | --- |
| SenseVoiceSmall | 安装包内置 | 默认语音输入，中文和低延迟优先 |

## 当前实现建议

1. 文件转写使用 `sherpa-onnx-offline.exe`；实时语音输入使用 Sherpa-ONNX Rust API 和 Silero VAD，不需要 WebSocket 服务端。
2. SenseVoice、Qwen3 INT8 ONNX、Silero VAD、Sherpa-ONNX 和 FFmpeg 都在构建阶段准备并打进安装包。
3. 构建脚本校验固定模型 SHA-256 和固定版本运行时归档；资源不完整时构建会直接失败。
4. 文件转写时，WAV 可直接送入 Sherpa-ONNX；MP3、M4A、MP4 等格式需要本机 ffmpeg 转成 16kHz 单声道 WAV。
5. 默认模型从当前安装资源目录动态解析，用户不需要选择或修复路径；高级用户仍可选择已有本地模型目录。
6. Qwen3-ASR 和 SenseVoice 读取配置时均以 `engine.json` 所在目录为准，因此模型文件夹可以整体复制。

## GPU 加速状态

当前正式路线是 Sherpa-ONNX CPU。CUDA 产品入口已移除，原因是 Windows 分发需要额外匹配 CUDA Toolkit、cuDNN、ONNX Runtime CUDA 依赖，安装包体积和用户环境不确定性都偏高。

GPU inference is not part of the current product. Recognition uses the bundled Sherpa-ONNX CPU runtime.

### Qwen3-ASR 0.6B 加速结论

- Qwen3 高效版使用 INT8 ONNX 权重和 Sherpa-ONNX CPU 推理，不启动本地 HTTP 服务。
- Vulkan 全卸载约 14.2 秒，CPU 音频编码加 Vulkan 文本解码约 6.1 秒，均慢于纯 CPU，因此正式高效版不包含 GPU 运行组件。
- Qwen3-ASR INT8 官方归档约 838 MiB，安装后保留 ONNX 模型、tokens 和 tokenizer 目录。
